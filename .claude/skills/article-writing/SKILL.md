# GBC Articles 記事執筆ワークフロー

メインサイトの 11ty（Eleventy）記事管理システム（`/articles/` 以下）で記事を作成・編集・リライトするための必須ワークフロー。

## ⚠️ 最初に必ず読むこと（references/）

記事の下書きを書き始める前に、このフォルダ内の以下の3ファイルを必ず読むこと：

| ファイル | 内容 | 読むタイミング |
|---------|------|--------------|
| `references/writing-style.md` | 文体ガイドライン（Kikumiさんの文体・答えリード・敬称・アンバサダー視点） | 全記事で必須 |
| `references/seo-aeo-rules.md` | SEO・AEO記事作成指針（店名表記・タイトル・キーワード・FAQの書き方） | 全記事で必須 |
| `references/pillar-articles.md` | 解説記事（ピラー記事）の戦略・テンプレート・推奨テーマリスト | ピラー記事（〜とは？系）を書く場合に必須 |

## ファイル構成

```
src/
├── articles/                         # 記事ソース（Markdownで追加）
│   ├── *.md                          # 個別記事ファイル
│   ├── index.njk                     # 記事一覧ページ
│   ├── articles.json.11ty.js         # 記事リストJSON（自動生成）
│   ├── tags/index.njk                # カテゴリ別一覧（自動生成）
│   └── articles-sitemap.xml.11ty.js  # 記事サイトマップ（自動生成）
├── _layouts/
│   ├── articles-base.njk             # 記事ページ共通ナビ・フッター
│   └── article.njk                   # 個別記事テンプレート（サイドバー付き）
.eleventy.js                          # 11ty設定ファイル
```

## ビルドコマンド

```bash
npm run dev    # ローカル確認（記事ページのみ。検索機能は動作しない）
npm run build  # 本番ビルド（eleventy + pagefind。Cloudflare Pagesが自動実行）
```

**重要**: `npm run dev` では検索（Pagefind）は動作しない。検索テストは `npm run build` 後に `npx serve public` で確認。

## 記事執筆開始時の確認事項（必須）

「記事を書きたい」という意図が伝わったら、5つの質問の**前に必ず**以下を問いかけること：

> 作業を始める前に、以下のコマンドで最新の状態に更新してください：
> ```
> git pull origin main
> ```
> 完了したら教えてください。

ユーザーが完了を確認したら、**重複チェック**を行ってから5つの質問に進む。

## ⚠️ 記事重複チェック（必須・5つの質問の前に実施）

記事を書く前に、以下のコマンドで既存タイトルを確認すること：

```bash
grep -h "^title:" src/articles/*.md | sort
```

### 重複の判断基準

以下の条件に該当する場合は「重複」と見なす：

- 同じ主題（例：「打ち合わせ」「焙煎体験」「フォカッチャ」）を同じ切り口で書いている
- タイトルのキーワードが既存記事と3語以上重なる
- 記事の主な問い（「〜するなら？」「〜はどこ？」）が既存記事と実質同じ

### 重複が疑われる場合の対応

1. **別の角度を提案する** → ユーザーに「〔既存記事タイトル〕と似た内容になりますが、〔別角度〕で書くのはいかがですか？」と提案
2. **既存記事の更新を提案する** → 追加したい情報が少量の場合、既存記事への加筆を提案
3. **ユーザーが「同じ内容でいい」と言った場合のみ** → 書き進める（サブテーマや固有キーワードを意識的にずらす）

### 現時点で記事数が多いテーマ（特に注意）

| テーマ | 既存記事数 | 書く場合の必須条件 |
|--------|-----------|------------------|
| ビジネス・打ち合わせ・仕事 | 9記事以上 | 「業種・シーン・感情」のどれかを新しく設定する |
| 焙煎体験・イベント | 6記事以上 | 「豆の種類・季節・学び」など固有の視点を設ける |
| 美容健康・コーヒー美容 | 5記事以上 | 「成分・食材・ライフスタイル」で未カバーの切り口にする |
| サードプレイス概念 | 8記事以上 | 「人物像・曜日・感情の変化」など具体的なシーンで差別化 |
| グルテンフリーフォカッチャ | 2記事（要整理） | 原則新記事を書かず既存記事の更新を優先 |

## 5つの質問（必須）

記事を書き始める前に、必ず以下をユーザーに確認すること：

```
1. カテゴリ：以下の10つから選んでください（原則1つ、内容が跨る場合は最大2つまでOK）
   コーヒー / 焙煎 / グルテンフリー / 体験・イベント / ペット / 美容健康 / ビジネス / サードプレイス / カフェ店内 / ダガヤサンドウ

2. 書きたいこと・伝えたいこと：メモ程度でOKです

3. まとめセクションをつけますか？（つける / つけない）

4. 記事末尾にQ&Aをつけますか？（つける / つけない）※AEO対策のため「つける」を推奨

5. 画像：提供しますか？Unsplashで探しますか？
```

## カテゴリ（固定・追加不可）

- コーヒー
- 焙煎
- グルテンフリー
- 体験・イベント
- ペット
- 美容健康
- ビジネス
- サードプレイス
- カフェ店内
- ダガヤサンドウ

**注意**: カテゴリはこの10つに固定。新カテゴリは追加しないこと。記事を書く際は必ずこの中から選ぶ。追加・変更はユーザーの明示的な指示があった場合のみ行う。

**複数カテゴリのルール**:
- 原則1カテゴリー。内容が自然に跨る場合のみ最大2つまで選択可
- 3つ以上選びたくなったら「記事の切り口を絞る」サインと考え、ユーザーに確認する
- 複数選んだ場合、フィルターのどちらのボタンを押しても記事が表示される（現システムで対応済み）

## フロントマターのテンプレート

**通常の記事（体験記・コラム）:**
```markdown
---
layout: article.njk
title: 記事タイトル
description: 100〜120文字の要約
date: 2026-05-07
tags:
  - articles
  - カテゴリ名
thumbnail: /assets/images/hero_bg-1.jpg
---
```

**ピラー記事（定義・解説ガイド）:**
```markdown
---
layout: article.njk
title: 〇〇とは？タイトル
description: 100〜120文字の要約
tags:
  - articles
  - カテゴリ名
pillar: true
thumbnail: /assets/images/articles/〇〇.jpg
---
```

**⚠️ 注意（過去に繰り返し違反あり・2026-07に11本全滅していたことが発覚）**: ピラー記事には `date` フィールドを**絶対に入れない**。日付を入れると「情報が古い」と判断されやすくなるため、常時有効なガイド記事としての性質を保つ。`pillar: true` がある記事に `date:` を見つけたら即削除すること。

**確認コマンド（ピラー記事に触れる作業のたびに実行すること）**:
```bash
grep -l "pillar: true" src/articles/*.md | xargs grep -l "^date:"
```
このコマンドで1件でもヒットしたら即座に該当行を削除する。「注意書きを読んだから大丈夫」ではなく、機械的に確認することを徹底する（文章の注意書きだけでは2026-07に11本すべてで見落とされた実績があるため）。

`pillar: true` を追加すると記事一覧ページとメインサイトのArticlesセクションに「GBCガイド」として特別表示される。

## 更新日（updated）フィールドのルール

記事を大きくリライトしたときは、フロントマターに `updated: YYYY-MM-DD` を追加する（2回目以降は日付を書き換え）。

```markdown
---
layout: article.njk
title: 記事タイトル
description: 100〜120文字の要約
date: 2026-05-07
updated: 2026-07-11
tags:
  - articles
  - カテゴリ名
thumbnail: /assets/images/hero_bg-1.jpg
---
```

**反映される場所（`updated` を追加すると自動で連動する）：**
- JSON-LD `BlogPosting` の `dateModified`（`src/_layouts/articles-base.njk`）
- `public/articles-sitemap.xml` の `lastmod`
- 記事ページのメタ情報に読者向けの「更新日：」表示（`src/_layouts/article.njk`。`updated` がない記事には表示されない）

**⚠️ `updated` を追加してよい場合／悪い場合：**
- ◯ 記事本文を実質的に書き直した・情報を追加した（読者が読み直す価値がある変更）
- ✕ 誤字修正など軽微な変更
- ✕ 内部リンクを1本追加しただけの編集（⑤のピラーリンク強化作業など）

中身を伴わない変更で `updated` だけ書き換えると、Googleのスパムポリシーが禁止する「date spam」とみなされるリスクがあるため、上記の基準を守ること。

## 記事画像のルール

- 記事用画像の保存先: `public/assets/images/articles/`
- **画像の選択元**: ユーザーが「画像を提供する」と言った場合、`public/assets/images/articles/` フォルダの中から**更新日時が最新のファイル**をサムネイルとして使用する
  - 確認コマンド: `ls -lt public/assets/images/articles/ | head -5`
  - 最上位に表示されるファイルが最新画像
- 画像はSEO的により説明的なファイル名に**リネームする**（例: `02.jpg` → `business-latte-dagayasando-gbc.jpg`）
- リネームは記事作成と同じタイミングで行い、フロントマターのパスも同時に更新する
- **サムネイル画像の配置**: 記事一覧・カテゴリ一覧のサムネイルは必ず `object-center`（上下中央）を使用すること。`object-top` は使わない
- **サムネイル画像コンテナ**: 固定高さ（`h-28` など）は使わず `self-stretch` を使うこと。固定高さだとカードの高さより画像コンテナが短くなり上寄りに見える

画像が提供された場合は、ファイルサイズを確認すること。**200KB超の場合は保存前に以下を案内する：**

> 画像サイズが大きいため、以下のサービスで圧縮してから再度お渡しください（目安：200KB前後）。
> - **TinyPNG** https://tinypng.com/ （PNG・JPEG対応）
> - **Squoosh** https://squoosh.app/ （Googleのツール）
> - **Optimizilla** https://imagecompressor.com/ （複数画像の一括処理可）

### thumbnailAlt（アイキャッチのalt属性・必須）

- フロントマターに `thumbnailAlt` を追加し、画像に何が写っているかを**30〜60字程度の自然な日本語**で書く（`src/_layouts/article.njk` が対応。未設定の場合は「記事タイトル | Green Beans Coffee 北参道・千駄ヶ谷」に自動フォールバックする）
- キーワードの羅列ではなく「画像の内容の説明」にする（例: ✕「北参道 カフェ ビジネス」→ ◯「ダガヤサンドウのカフェで提供されるカフェラテ、ビジネス利用のイメージ」）
- 新規記事作成時は`thumbnail`と同じタイミングで必ず設定する

```markdown
thumbnail: /assets/images/articles/business-latte-dagayasando-gbc.jpg
thumbnailAlt: ダガヤサンドウのカフェで提供されるカフェラテ、ビジネス利用のイメージ
```

### 画像の使い回し禁止（新規記事・特にピラー記事で必須）

- **同じ画像ファイルを複数の記事の`thumbnail`に指定しないこと**。1つの画像URLが複数ページで使われるとGoogle画像検索が1記事にしかインデックスしない場合があり、SNSシェア時にも記事の見分けがつきにくくなる
- 新規記事で使う予定の画像が既存記事で使用済みの場合は、その画像ファイルを**コピーしてその記事専用のファイル名にリネーム**してから使う（実写真は同じでも、URL・ファイル名・altは記事ごとに一意にする）
  - リネーム例: 記事slugを使って `<記事slug>-gbc.jpg` とする
- 特にピラー記事は「そのテーマの代表記事」という位置づけのため、専用画像を必ず用意する。2026-07にコピー＆リネームで応急処置していた11本すべてをユーザー提供の実写真に差し替え完了。以後ピラー記事の画像は使い回しコピーではなく実写真を用意することを原則とする
- ユーザーから写真が届いたら、まずどのピラー記事・タグに対応する内容かを画像の被写体から確認し、記事のテーマと合っているか一言で確認してから反映する（過去に被写体違いの写真が別記事宛てに送られてきたケースがあったため）

## 記事FAQ（よくある質問）の書式ルール

Markdownでは1行の改行が無視されるため、**Q.**タイトルと回答が同じ行に連結して表示される。必ず `<br>` を使って明示的に改行すること。

```markdown
**Q. 質問文**<br>
回答文。
```

❌ 誤り（タイトルと回答がくっつく）:
```markdown
**Q. 質問文**
回答文。
```

✅ 正しい:
```markdown
**Q. 質問文**<br>
回答文。
```

## 記事ページのメタタグ・JSON-LDルール

### description の文字数（100〜120字厳守）
- **100字未満**: 検索結果での情報量が不足する → 短すぎはNG
- **120字超**: 検索結果で切り捨てられる → 長すぎもNG
- **対象は3つすべて**: `name="description"` / `og:description` / `twitter:description`
- 新記事作成後・既存記事編集後は文字数を必ず確認すること
- **⚠️ 締め文を入れない（文字数超過の主な原因）**: 「北参道GBCに通い続けた経験をもとにお伝えします」「〜の経験をもとにお伝えします」などの締め文はdescriptionに入れない。定義文＋場所情報＋体験内容で構成し、締め文は省く
- **descriptionの推奨構成（ピラー記事）**: 「〇〇とは△△のことです。（定義文）」＋「東京・北参道のGreen Beans Coffeeで〜できます。（場所と体験）」で100〜120字に収める

### og:type
- **個別記事ページ**: `article`
- **記事一覧・タグ一覧・LPページ（メイン・ペット）**: `website`
- `articles-base.njk` では `ogType` 変数で動的に切り替えている。一覧・タグページのフロントマターには必ず `ogType: website` を記述すること

### Speakable セレクタ（記事ページ）
- `src/_layouts/articles-base.njk` の `cssSelector` は `["h1", "h2", "article .prose > p:first-of-type"]` を維持する
- `p:first-of-type` = 答えリード（冒頭太字段落）を音声検索対象として指定
- `article .prose p`（全段落）は広すぎるため使わない

### 記事著者情報（動的展開）
- 記事frontmatterに `author` フィールドを追加すると著者名が動的にJSON-LDへ反映される
- 省略した場合はデフォルト `Kikumi` が使われる
- ペット記事など別著者が書く場合は `author: Ayaka` のように明記する

## 内部リンク（AEO対策・毎回必須）

新記事を作成するたびに、以下の2つを必ず行うこと：

1. **新記事 → 既存記事へのリンク（3〜4本）**
   - まとめセクションの直前に、関連する既存記事へのアンカーテキストリンクを自然な文章で挿入する
   - リンクは箇条書きではなく、文章の中に自然に埋め込む

2. **既存記事 → 新記事への逆リンク（2〜3本）**
   - 新記事と関連する既存記事を選び、その記事の適切な箇所に新記事へのリンクを追記する
   - 既存記事のまとめ直前など、文脈に合う場所に1文追加する形で挿入する

3. **新記事 → 対応するピラー記事へのリンク（最低1本）**
   - 新記事のタグから下表で対応するピラー記事を判定し、本文中（まとめ直前が自然）に1文リンクを追加する
   - 同じキーワードを狙う記事が増えるたびにピラー記事への被リンクが薄まりカニバリゼーションが起きやすくなるため、新規記事の作成時点で毎回リンクしておくことで防止する
   - 既にそのピラーへのリンクが本文にある場合は追加不要

   | タグ | 対応するピラー記事 |
   |------|-------------------|
   | 焙煎 | roasting-machine-guide |
   | コーヒー（タイトルに「ハンドドリップ」を含む場合） | handdrip-guide |
   | ビジネス | business-cafe-guide |
   | ペット | pet-cafe-guide |
   | グルテンフリー | gluten-free-guide |
   | 美容健康 | coffee-beauty-health-guide |
   | サードプレイス | thirdplace-guide |
   | ダガヤサンドウ | dagayasando-guide |
   | コーヒー（上記に該当しない場合） | specialty-coffee-guide |
   | カフェ店内 | cafe-interior-guide |
   | 体験・イベント（焙煎タグがない場合） | event-guide |

   複数タグに該当する場合は表の上から優先する。判定に迷う場合や該当ピラーがない場合はユーザーに確認する。

ユーザーからの指示がなくても、記事作成の仕上げとして自動的に実施すること。

**注意**: 2026-07時点で既存記事53本すべてが上記のいずれかのピラーへリンク済み（⑤カニバリゼーション対策の一環として一括対応済み）。今後はこのルールにより新規記事の追加時点で被リンクが分散しないようにする。

## 記事の下書き完成後（必須）：カテゴリーの評価と報告

下書きをユーザーに渡す前に、**Claudeが記事の内容を読んでカテゴリーの適切さを自己評価すること**。単なる確認ではなく、問題があれば理由を添えて指摘する。

**評価の観点（すべてチェックすること）：**

1. **主題の一致** — カテゴリーが「記事の主題」を表しているか。舞台（GBCというコーヒー店）ではなく、記事が何について書かれているかで判断する
2. **ダブりによる焦点のボケ** — 複数カテゴリーの場合、それぞれが記事の中で実質的な比重を持っているか。片方が「ついでに触れた程度」なら外す
3. **コーヒーの過剰付与** — 「GBCの記事だからコーヒー」という理由だけでコーヒーカテゴリーをつけていないか。コーヒーそのもの（味・豆・抽出・品質）が主題でない限りつけない
4. **より適切なカテゴリーの存在** — 選ばれたカテゴリーより正確なものが10個の中にないか（例：コーヒー全般より「焙煎」のほうが正確、など）

**下書きと一緒に渡すメッセージの形式：**

問題がない場合：
> 📌 **カテゴリー：〔カテゴリー名〕**
> 記事の主題と一致しています。このまま保存します。

変更を提案する場合：
> 📌 **カテゴリーの変更を提案します。**
> 当初の選択：〔元カテゴリー〕
> 提案：〔新カテゴリー〕
> 理由：〔具体的な理由。例：「コーヒーに触れているのはペアリングの一文だけで、記事の主題は食事なので外すほうが焦点が明確になります」〕
> このまま変更してよいですか？

- カテゴリーが確定してから `frontmatter` に記入してファイルを保存する
- ユーザーが最初に選んだカテゴリーを尊重しつつ、完成した記事の内容と照合して判断する

---

## 自動生成されるファイル（.gitignore対象）

- `public/articles/` — ビルド時に生成
- `public/articles-sitemap.xml` — 記事サイトマップ（記事追加時に自動更新）
- `public/pagefind/` — 検索インデックス

# プロジェクトの名前
アンバサダー Kikumiによるグリーンビーンズコーヒーのファンサイト

## プロジェクト概要

- 非エンジニア、非デザイナーが、AI Agent (Claude Code)を使ってWebサイトを構築する
- **AEO（Answer Engine Optimization）対策を最優先**とする

## サイト基本情報

- **タイトル**: Green Beans Coffee
- **サブタイトル**: 店名看板のないペットフレンドリーカフェ
- **アンバサダー**: Kikumi（Heroセクション右下に「GBCアンバサダー Kikumi」と表示）

### 用語統一
- 「**店名看板のないカフェ**」で統一すること。「看板のないカフェ」は誤り。
- 「**新宿から10分ほど**」で統一すること。「新宿から電車で10分ほど」は誤り（「電車で」は不要）。
- 「**千駄ヶ谷駅徒歩7分**」で統一すること。「徒歩5分」「徒歩圏内」は誤り。北参道駅は「徒歩3分」で統一。
  - ⚠️ この表記ルールは **GBCへのアクセス（最寄り駅からの徒歩時間）** にのみ適用する。国立競技場・鳩森八幡神社・明治神宮など **周辺スポットへの距離**は対象外（「鳩森八幡神社まで徒歩5分」等はOK）。
- **ダガヤサンドウの方位**（GBC・北参道駅基準）：北側＝代々木・新宿、北東側＝新宿御苑・東京体育館、東側＝国立競技場、南東側＝外苑前・青山、南側＝原宿・渋谷、南西側＝明治神宮・代々木公園。「国立競技場・東京体育館が北側」「外苑前・青山が東側」「明治神宮・代々木公園が西側」「代々木・新宿が北西側」は誤り。
- **GBC体験イベントは2種類**：「ハンドドリップ体験」と「焙煎体験（オリジナルブレンド焙煎含む）」。「焙煎体験」と「オリジナルブレンド焙煎体験」は同じイベント。「3種類」と書かないこと。


# 作成のためのルール、ワークフロー

## 保存先

- ファイルを作成する際は、必ずUTF-8エンコーディングで保存し、日本語の文字化けが発生しないよう注意してください
- public ディレクトリに成果物を保存する
- 特にユーザーの指示がなければ、index.html で作成する
- 指示があれば、指示に合わせたファイルを publicの下に保存する
- 画像、CSS、Javascriptなどを生成した場合は、public/assets/ 以下の所定のフォルダに保存する

## あなたの仕事

### ⚠️ Git操作の重要ルール（絶対厳守）

**🚨 絶対に守ること 🚨**

1. **ユーザーの明確な指示があるまで、コミットもプッシュも絶対に行わない**
2. **コミットの指示があった場合** → コミットのみ実行し、完了を報告。プッシュはしない
3. **プッシュの指示があった場合** → その時初めてプッシュを実行
4. **「コミット＆プッシュ」の指示があった場合のみ** → 両方を実行

**禁止事項:**
- ❌ 勝手にコミット＆プッシュを連続実行する
- ❌ 「コミットしました」と言って勝手にプッシュもする
- ❌ ユーザーの確認なしに任意のGit操作を行う
- ❌ 作業完了後に自動でコミット＆プッシュする

**このルールは何度も破られているため、各作業の完了時に必ず「コミット＆プッシュの指示をお待ちしております」と伝え、ユーザーの指示を待つこと。**

---

### ⚠️ ブランチ戦略（必須）

**基本方針:**

- **メインサイトとペットサイトは両方とも `main` ブランチで管理**
- 日常的な更新（FAQ追加、コンテンツ更新、画像差し替え等）は直接 `main` ブランチで実施
- 大きな機能追加や実験的な変更のみ、短期的なフィーチャーブランチを使用
  - 例: 新しいセクション追加、大規模リニューアル、デザイン変更等
  - 完成したらすぐに `main` にマージしてブランチを削除

**作業時の注意:**

1. **変更前の確認** → `git status` で未コミットの変更があるか確認
2. **ユーザーの手動編集を尊重** → ユーザーが直接ファイルを編集している可能性を常に考慮
3. **フィーチャーブランチを使う場合** → 必要に応じて `git stash` で一時保存してからブランチを切り替え
4. **ブランチ切り替え時** → 変更が失われないよう、必ず stash または commit してから切り替えること

### 📌 タグ付けルール

**命名規則**: `YYYY.MM.パッチ番号` の形式（カレンダーバージョニング）

**例**:
```
2026.02.0 - 2月号の初回リリース（1月最終週に作業）
2026.02.1 - 2月号への追加修正
2026.02.2 - 2月号への2回目の修正
```

**運用ルール**:
- タグの月番号は「対象コンテンツの月」を示す（実際の作業月ではない）
- GBCマガジンの月号更新など、翌月コンテンツを追加する場合は翌月のタグ番号を使う
- 同じ月のコンテンツに対する修正は、パッチ番号をインクリメント（例: 2026.02.0 → 2026.02.1）
- コンテンツと無関係な修正（FAQ誤字など）は、現在の月のタグを使う

**タグの移動**:
- タグを間違えた位置につけた場合は、削除して再作成できる
- コマンド例:
  ```bash
  git tag -d 2026.02.0
  git push origin :refs/tags/2026.02.0
  git tag -a 2026.02.0 -m "メッセージ"
  git push origin 2026.02.0
  ```

## 画像の取得

- ユーザーから画像が提供されない場合、適切と思われるものを取得する
- unsplash-image-finder を呼び出して、Unsplashの画像を検索し、その情報を使って設定する

## 記事画像のルール

- 記事用画像の保存先: `public/assets/images/articles/`
- **画像の選択元**: ユーザーが「画像を提供する」と言った場合、`public/assets/images/articles/` フォルダの中から**更新日時が最新のファイル**をサムネイルとして使用する
  - 確認コマンド: `ls -lt public/assets/images/articles/ | head -5`
  - 最上位に表示されるファイルが最新画像
- 画像はSEO的により説明的なファイル名に**リネームする**（例: `02.jpg` → `business-latte-dagayasando-gbc.jpg`）
- リネームは記事作成と同じタイミングで行い、フロントマターのパスも同時に更新する
- **サムネイル画像の配置**: 記事一覧・カテゴリ一覧のサムネイルは必ず `object-center`（上下中央）を使用すること。`object-top` は使わない
- **サムネイル画像コンテナ**: 固定高さ（`h-28` など）は使わず `self-stretch` を使うこと。固定高さだとカードの高さより画像コンテナが短くなり上寄りに見える

## 画像の最適化について

- 画像の最適化が必要であれば、ユーザーに伝える
- 画像最適化については、以下の無料サービスを案内する:
    - **TinyPNG** (https://tinypng.com/): PNG・JPEG最適化の定番
    - **Squoosh** (https://squoosh.app/): Googleが提供する画像圧縮ツール
    - **Optimizilla** (https://imagecompressor.com/): 複数画像の一括処理対応

  - Primary: #4A3728（ダークブラウン）
  - Secondary: #8B7355（ミディアムブラウン）

### レスポンシブデザイン原則

**テキストの配置ルール（2文以上のテキストブロック）:**

複数文で構成されるテキストブロック（リード文、説明文など）は、以下のルールに従う：

- **モバイル表示**: 改行なしで左寄せ（`text-left`）
- **デスクトップ表示**: 適切な位置で改行し中央寄せ（`md:text-center`）
- **実装方法**: `text-left md:text-center` + `<br class="hidden md:block">`

**適用例:**
```html
<p class="text-left md:text-center text-lg mb-16 max-w-3xl mx-auto">
  最初の文章。<br class="hidden md:block">次の文章。
</p>
```

**適用箇所:**
- 各セクションのリード文（導入文）
- 複数文で構成される説明文や紹介文
- その他、2文以上のテキストブロック

**重要**: このルールは一貫性を保つため、サイト全体で徹底すること。

# AEO対策 重要事項

## JSON-LD構造化データ（必須）

以下の構造化データを `<head>` 内または `</body>` 直前に設置:

1. **LocalBusiness (CafeOrCoffeeShop)**: 店舗情報
2. **FAQPage**: よくある質問（メインサイト: 19問、ペットサイト: ペット特化FAQ）
3. **WebSite**: サイト情報

詳細は `project-docs/technical-spec.md` を参照。

## メタタグ・OGP実装ルール（必須）

### og:type
- **個別記事ページ**: `article`
- **記事一覧・タグ一覧・LPページ（メイン・ペット）**: `website`
- `articles-base.njk` では `ogType` 変数で動的に切り替えている。一覧・タグページのフロントマターには必ず `ogType: website` を記述すること

### favicon
- パスは必ず**絶対パス** `/favicon.png` で記述する
- 相対パス `favicon.png` はサブディレクトリ（`/pets/` 等）で読み込まれないため使用禁止

### twitter:description と og:description
- 両者は**必ず同一の内容**にすること。別の文章を書かない
- `twitter:description` だけ独自の文章にすると一貫性が崩れてSNS共有時に混乱が生じる

### description の文字数（100〜120字厳守）
- **100字未満**: 検索結果での情報量が不足する → 短すぎはNG
- **120字超**: 検索結果で切り捨てられる → 長すぎもNG
- 新記事作成後・既存記事編集後は文字数を必ず確認すること
- **⚠️ 締め文を入れない（文字数超過の主な原因）**: 「北参道GBCに通い続けた経験をもとにお伝えします」「〜の経験をもとにお伝えします」などの締め文はdescriptionに入れない。定義文＋場所情報＋体験内容で構成し、締め文は省く
- **descriptionの推奨構成（ピラー記事）**: 「〇〇とは△△のことです。（定義文）」＋「東京・北参道のGreen Beans Coffeeで〜できます。（場所と体験）」で100〜120字に収める

## FAQセクション

- アコーディオン形式で実装
- **メインサイト**: 19問の基本Q&Aを掲載
- **ペットサイト**: ペット特化FAQを掲載（メインサイトとは独立）
- JSON-LD FAQPage と連動

## 記事FAQ（よくある質問）の書式ルール

Markdownでは1行の改行が無視されるため、**Q.**タイトルと回答が同じ行に連結して表示される。必ず `<br>` を使って明示的に改行すること。

```markdown
**Q. 質問文**<br>
回答文。
```

❌ 誤り（タイトルと回答がくっつく）:
```markdown
**Q. 質問文**
回答文。
```

✅ 正しい:
```markdown
**Q. 質問文**<br>
回答文。
```

## コンテンツの書き方

- 自然で人間味のある文章（AIが引用しやすい）
- 具体的な情報を明記（住所、電話番号、営業時間など）
- 「北参道」「千駄ヶ谷」「ダガヤサンドウ」などの地域名を自然に含める
- ペットフレンドリー、グルテンフリー、コーヒー専門店などのキーワードを含める
- 記事には「東京カフェ」「スペシャルティコーヒー」も文脈に合う範囲で自然に含める

# GBC Articles（11ty 記事管理）

## 概要

メインサイトに 11ty（Eleventy）を使った記事管理システムを追加。`/articles/` 以下で動作。

## ファイル構成

```
src/
├── articles/                         # 記事ソース（Markdownで追加）
│   ├── *.md                          # 個別記事ファイル
│   ├── index.njk                     # 記事一覧ページ
│   ├── articles.json.11ty.js         # 記事リストJSON（自動生成）
│   ├── tags/index.njk                # カテゴリ別一覧（自動生成）
│   └── articles-sitemap.xml.11ty.js  # 記事サイトマップ（自動生成）
├── _layouts/
│   ├── articles-base.njk             # 記事ページ共通ナビ・フッター
│   └── article.njk                   # 個別記事テンプレート（サイドバー付き）
.eleventy.js                          # 11ty設定ファイル
```

## ビルドコマンド

```bash
npm run dev    # ローカル確認（記事ページのみ。検索機能は動作しない）
npm run build  # 本番ビルド（eleventy + pagefind。Cloudflare Pagesが自動実行）
```

**重要**: `npm run dev` では検索（Pagefind）は動作しない。検索テストは `npm run build` 後に `npx serve public` で確認。

## フロントマターのテンプレート

**通常の記事（体験記・コラム）:**
```markdown
---
layout: article.njk
title: 記事タイトル
description: 100〜120文字の要約
date: 2026-05-07
tags:
  - articles
  - カテゴリ名
thumbnail: /assets/images/hero_bg-1.jpg
---
```

**ピラー記事（定義・解説ガイド）:**
```markdown
---
layout: article.njk
title: 〇〇とは？タイトル
description: 100〜120文字の要約
tags:
  - articles
  - カテゴリ名
pillar: true
thumbnail: /assets/images/articles/〇〇.jpg
---
```

**⚠️ 注意（過去に繰り返し違反あり）**: ピラー記事には `date` フィールドを**絶対に入れない**。日付を入れると「情報が古い」と判断されやすくなるため、常時有効なガイド記事としての性質を保つ。`pillar: true` がある記事に `date:` を見つけたら即削除すること。

`pillar: true` を追加すると記事一覧ページとメインサイトのArticlesセクションに「GBCガイド」として特別表示される。

## メインサイトのGBCガイド表示ルール

- **常に最新ピラー記事2件を表示する**（`public/index.html` の `#articles` セクション内）
- 新しいピラー記事を追加したら、`public/index.html` の GBCガイドカード2枚を最新2件に差し替えること
- 対象箇所: `<!-- GBCガイド（ピラー記事） -->` のコメント以下 `grid grid-cols-1 sm:grid-cols-2` のグリッド内
- 更新時は thumbnail・title・description・href を新記事に合わせて変更する

## カテゴリ（固定・追加不可）

- コーヒー
- 焙煎
- グルテンフリー
- 体験・イベント
- ペット
- 美容健康
- ビジネス
- サードプレイス
- カフェ店内
- ダガヤサンドウ

**注意**: カテゴリはこの10つに固定。新カテゴリは追加しないこと。記事を書く際は必ずこの中から選ぶ。追加・変更はユーザーの明示的な指示があった場合のみ行う。

**複数カテゴリのルール**:
- 原則1カテゴリー。内容が自然に跨る場合のみ最大2つまで選択可
- 3つ以上選びたくなったら「記事の切り口を絞る」サインと考え、ユーザーに確認する
- 複数選んだ場合、フィルターのどちらのボタンを押しても記事が表示される（現システムで対応済み）

## 記事執筆開始時の確認事項（必須）

## ⚠️ 記事重複チェック（必須・5つの質問の前に実施）

記事を書く前に、以下のコマンドで既存タイトルを確認すること：

```bash
grep -h "^title:" src/articles/*.md | sort
```

### 重複の判断基準

以下の条件に該当する場合は「重複」と見なす：

- 同じ主題（例：「打ち合わせ」「焙煎体験」「フォカッチャ」）を同じ切り口で書いている
- タイトルのキーワードが既存記事と3語以上重なる
- 記事の主な問い（「〜するなら？」「〜はどこ？」）が既存記事と実質同じ

### 重複が疑われる場合の対応

1. **別の角度を提案する** → ユーザーに「〔既存記事タイトル〕と似た内容になりますが、〔別角度〕で書くのはいかがですか？」と提案
2. **既存記事の更新を提案する** → 追加したい情報が少量の場合、既存記事への加筆を提案
3. **ユーザーが「同じ内容でいい」と言った場合のみ** → 書き進める（サブテーマや固有キーワードを意識的にずらす）

### 現時点で記事数が多いテーマ（特に注意）

| テーマ | 既存記事数 | 書く場合の必須条件 |
|--------|-----------|------------------|
| ビジネス・打ち合わせ・仕事 | 9記事以上 | 「業種・シーン・感情」のどれかを新しく設定する |
| 焙煎体験・イベント | 6記事以上 | 「豆の種類・季節・学び」など固有の視点を設ける |
| 美容健康・コーヒー美容 | 5記事以上 | 「成分・食材・ライフスタイル」で未カバーの切り口にする |
| サードプレイス概念 | 8記事以上 | 「人物像・曜日・感情の変化」など具体的なシーンで差別化 |
| グルテンフリーフォカッチャ | 2記事（要整理） | 原則新記事を書かず既存記事の更新を優先 |

## 記事の下書き完成後（必須）：カテゴリーの評価と報告

下書きをユーザーに渡す前に、**Claudeが記事の内容を読んでカテゴリーの適切さを自己評価すること**。単なる確認ではなく、問題があれば理由を添えて指摘する。

**評価の観点（すべてチェックすること）：**

1. **主題の一致** — カテゴリーが「記事の主題」を表しているか。舞台（GBCというコーヒー店）ではなく、記事が何について書かれているかで判断する
2. **ダブりによる焦点のボケ** — 複数カテゴリーの場合、それぞれが記事の中で実質的な比重を持っているか。片方が「ついでに触れた程度」なら外す
3. **コーヒーの過剰付与** — 「GBCの記事だからコーヒー」という理由だけでコーヒーカテゴリーをつけていないか。コーヒーそのもの（味・豆・抽出・品質）が主題でない限りつけない
4. **より適切なカテゴリーの存在** — 選ばれたカテゴリーより正確なものが9つの中にないか（例：コーヒー全般より「焙煎」のほうが正確、など）

**下書きと一緒に渡すメッセージの形式：**

問題がない場合：
> 📌 **カテゴリー：〔カテゴリー名〕**
> 記事の主題と一致しています。このまま保存します。

変更を提案する場合：
> 📌 **カテゴリーの変更を提案します。**
> 当初の選択：〔元カテゴリー〕
> 提案：〔新カテゴリー〕
> 理由：〔具体的な理由。例：「コーヒーに触れているのはペアリングの一文だけで、記事の主題は食事なので外すほうが焦点が明確になります」〕
> このまま変更してよいですか？

- カテゴリーが確定してから `frontmatter` に記入してファイルを保存する
- ユーザーが最初に選んだカテゴリーを尊重しつつ、完成した記事の内容と照合して判断する

---

画像が提供された場合は、ファイルサイズを確認すること。**200KB超の場合は保存前に以下を案内する：**

> 画像サイズが大きいため、以下のサービスで圧縮してから再度お渡しください（目安：200KB前後）。
> - **TinyPNG** https://tinypng.com/ （PNG・JPEG対応）

## SEO・AEO記事作成指針

**店名の表記ルール（SEO対策・必須）：**

記事内での店名は以下の3種類を使い分ける。検索エンジンとAIに「同じ店の複数の呼び方」として認識させることがSEO・AEO対策に有効。

| 表記 | 使い方 |
|-----|--------|
| **Green Beans Coffee** | メインの表記。記事冒頭・見出し・まとめ・Q&Aなど、重要な箇所で使う |
| **グリーンビーンズコーヒー** | 日本語検索ヒット用。本文中に自然な形で数回使う（1記事あたり3〜5回を目安） |
| **GBC** | 同一段落内での2回目以降の言及など、繰り返しを避けたい箇所に使う |

**適用ルール：**
- 1記事の中で3種類すべてを使う
- 段落の冒頭・文の主語になる箇所は `Green Beans Coffee` または `グリーンビーンズコーヒー` を優先
- `GBC` は略称として自然に使い、多用しない
- 不自然な詰め込みはしない。「グリーンビーンズコーヒーのグリーンビーンズコーヒーが…」のような繰り返しは避ける

**タイトル（SEO最重要項目）：**

記事タイトルはSEO対策の中で最も重要な要素。以下の3つのキーワードグループを**必ず自然な形でタイトルに含める**こと。

**① カテゴリワード（記事のカテゴリに対応するワードを1つ）：**
- コーヒー・カフェ / 焙煎 / グルテンフリー / 体験・イベント / ペット / 美容健康 / ビジネス / サードプレイス / ダガヤサンドウ
- 例：「北参道のコーヒー専門店で〜」「千駄ヶ谷のペット同伴カフェで〜」

**② 近隣地名（1〜2つ）：**
- 北参道 / 千駄ヶ谷 / 原宿 / 代々木 / 渋谷 / 表参道 / 青山
- タイトルに地名が入ることで「エリア検索」にヒットしやすくなる

**③ 地域施設・周辺スポット（自然に入れられる場合）：**
- 国立競技場 / 東京体育館 / 明治神宮 / 明治神宮外苑 / 新宿御苑 / 鳩森八幡神社 / 代々木公園
- 施設訪問後の立ち寄りスポットとして検索されやすくなる
- 無理に入れず、記事の内容と自然に結びつく場合のみ使用する

**タイトル作成のルール：**
- 「〜してみた」「〜に行ってきました」「〜が最高だった」など体験型タイトルはクリックされやすく、AIにも引用されやすい
- 30〜35文字以内を目安（長すぎると検索結果で切れる）
- 検索意図を想定して立てる（「北参道 コーヒー おすすめ」「千駄ヶ谷 ペット カフェ」など）
- **「東京」または「渋谷」をタイトルに自然に入れる** → 文脈に合う場合は「東京・北参道の〜」「渋谷・北参道の〜」など冒頭に置くと効果的。ただし無理に入れると不自然になるため、すべてのタイトルに必須ではない。自然に溶け込む場合のみ使う
- 下書きを書き始める前に、SEO/AEOを意識したタイトル案を2〜3個提示してユーザーに選んでもらう

**FAQ（Q&A）の書き方：**
- AIや音声検索が「そのまま読み上げられる」簡潔な回答を心がける
- 質問は実際に検索されうる形で立てる。例：
  - 「東京・北参道でおすすめのコーヒー専門店は？」
  - 「千駄ヶ谷でスペシャルティコーヒーが飲めるカフェは？」
  - 「ダガヤサンドウエリアのペットと入れるカフェは？」
- 質問文に「東京」「渋谷」を自然に含めるとSEO効果が高まる。ただし全問に入れると不自然になるため、1記事あたり2〜3問を目安に、文脈に合う箇所だけに使う
- 回答は2〜3文に収める。専門性と信頼性を感じさせる語尾（「〜です」「〜できます」）で締める
- 回答の冒頭でそのまま答えを述べる（前置き不要）。例：「Green Beans Coffeeがおすすめです。北参道駅から徒歩3分、自家焙煎のスペシャルティコーヒーを楽しめます。」

**「渋谷」「東京」「新宿」ワードの必須挿入（SEO・AEO戦略・最重要）：**

GBCは渋谷区に所在する。「渋谷」「東京」「新宿」は検索ボリュームが大きく先行優位性が高いため、全記事に必ず自然な形で入れること。

- **「渋谷」** → 全記事に必ず1回以上入れる。「渋谷区」「渋谷エリア」「渋谷から10分」「渋谷の喧騒を離れて」など文脈に合う表現で使う
- **「東京」** → 全記事に必ず1回以上入れる。「東京カフェ」「東京のサードプレイス」「東京・渋谷区」など自然に入れる
- **「新宿」** → 全記事に必ず1回以上入れる。「新宿から10分」「新宿御苑の近く」「新宿エリア」など文脈に合う表現で使う
- まとめ・FAQ回答・アクセス説明・リード文など、どこでも文脈に合う場所に挿入する
- 不自然な詰め込みは禁止。あくまで文章の流れの中に溶け込ませる

**隣接エリアキーワード（SEO・AEO対策）：**

東京都・渋谷区・原宿・表参道・代々木・新宿などの隣接エリアワードが少ない記事には自然に追加すること。

- 全記事の8割を目安に含める（無理に詰め込まない）
- FAQ回答・まとめ・アクセス説明など、文脈に合う場所に1〜2ワード程度
- 既存の文章に溶け込む形で追加し、不自然にならないよう注意する

**周辺観光・散策エリアキーワード（SEO・AEO対策）：**

鳩森八幡神社 / 国立競技場 / 東京体育館 / 明治神宮 / 明治神宮外苑 / 新宿御苑 / 神宮球場 / 代々木公園 / 将棋会館 / 国立能楽堂などの周辺スポットワードが少ない記事には自然に追加すること。

- 全記事の7割を目安に含める（無理に詰め込まない）
- FAQ回答・まとめ・アクセス説明など、文脈に合う場所に1〜2ワード程度
- 既存の文章に溶け込む形で追加し、不自然にならないよう注意する

**内部リンク（AEO対策・毎回必須）：**

新記事を作成するたびに、以下の2つを必ず行うこと：

1. **新記事 → 既存記事へのリンク（3〜4本）**
   - まとめセクションの直前に、関連する既存記事へのアンカーテキストリンクを自然な文章で挿入する
   - リンクは箇条書きではなく、文章の中に自然に埋め込む

2. **既存記事 → 新記事への逆リンク（2〜3本）**
   - 新記事と関連する既存記事を選び、その記事の適切な箇所に新記事へのリンクを追記する
   - 既存記事のまとめ直前など、文脈に合う場所に1文追加する形で挿入する

ユーザーからの指示がなくても、記事作成の仕上げとして自動的に実施すること。

## 解説記事（定義記事）戦略

### なぜ解説記事がSEO・AEOに強いか

「〜とは？」「〜の選び方」「〜の特徴」といった定義・解説記事は以下の理由で効果が大きい：

- **Googleのゼロクリック検索**に直接表示される（強調スニペット）
- **ChatGPT・Claude・Gemini**などAIが定義を引用しやすい形式
- **検索ボリュームが安定**していて時間が経っても読まれ続ける
- 体験記記事への**ハブ（集約ページ）**として内部リンクを集めやすい

100記事・200記事・300記事・500記事を目指す際、**体験記6：解説記4**の割合を意識して増やしていく。

### FAQの作り方（解説記事特化）

解説記事のFAQは「検索されやすい質問」を意識して立てる。特に以下のパターンが効果的：

- **定義質問**：「〜とは何ですか？」「〜の意味を教えてください」

- **選び方質問**：「〜を選ぶときのポイントは？」「おすすめの〜は？」
- **場所質問**：「東京で〜できる場所は？」「渋谷・北参道で〜は？」
- **効果質問**：「〜にはどんな効果がありますか？」「〜はなぜ良いのですか？」

**回答の書き方**：
- 冒頭の1文で直接答える（前置き不要）
- 2〜3文に収める
- 「〜です」「〜できます」「〜があります」で締める
- 専門用語は使ったあとすぐ言い換える

## 文体ガイドライン（Kikumiさんの文体）

**⚠️ 記事種別による「Kikumi」名前使用ルール（必須）**

| 記事種別 | 名前の使用 |
|---------|-----------|
| ピラー記事（`pillar: true`・〜とは？系） | ❌ 使用しない |
| 体験記（通常記事・dateあり） | ✅ 本文中に自然に入れる |

**ピラー記事：名前を使わない**

定義・解説記事は情報の正確さで評価されるため、個人名ではなく実績表現を使う。

- ❌「GBCアンバサダーKikumiが解説します」
- ✅「北参道GBCに通い続けた経験をもとにお伝えします」
- ✅「GBCアンバサダーが実際の体験をもとに発信しています」

**体験記：本文中に自然に「Kikumi」を入れる**

E-E-A-T対策・パーソナルブランディング・読者との信頼感のため、体験記には本文中に自然な形でKikumiの名前を入れること。

- ✅「私Kikumiが実際に体験してきました」
- ✅「GBCアンバサダーのKikumiです」
- ❌ 不自然に連呼しない（1記事あたり1回を目安）

---

**敬称ルール（必須）**
人物を指す語には必ず敬称をつけること。

- ✅「バリスタさん」（❌「バリスタ」呼び捨て禁止）
- ✅「お客さん」（❌「お客」呼び捨て禁止）
- スタッフ・客・職人など、人を指す単語はすべて同様に扱う

---

**大前提：アンバサダー視点で書く（店舗関係者ではない）**
記事を書く人はGBCのスタッフでも店長でもなく「外部のアンバサダー」。
店やスタッフに聞いて知った・体験して気づいた、という立場を常に保つこと。

- ✅「〜と教えていただきました」「〜とのことです」「〜と聞きました」
- ✅「バリスタさんから〜を学びました」「スタッフの方に確認したところ〜」
- ❌「〜しています」「〜を提供しています」（店の公式説明のような書き方）
- ❌「〜にこだわっています」（店の意図を断言する書き方）

**Kikumiはすでにすべて知っている（3年以上の通い歴）**
KikumiはGBCに3年以上通い続けているアンバサダーであり、メニュー・サービス・スタッフ・空間をすでによく知っている前提で書くこと。

- ❌「私もずっと気になっていました」「初めて知りました」「話題のお店に行ってみました」（初訪問のような書き方）
- ❌「〜とのことで驚きました」（基本的なサービスを知らなかったような表現）
- ✅「通い続けるなかで気づいたことがあります」「改めてこの良さを伝えたいと思いました」
- ✅ 初めての訪問を**過去の思い出**として語るのはOK（「初めて訪れたとき〜」）
- ✅ 新しい発見・深掘りで知ったことを伝える文脈はOK（「バリスタさんに改めて聞いてみると〜」）

**このルールはオーナーがClaudeと一緒に記事を作る場合も同じく徹底すること。**
下書き生成後は必ずアンバサダー視点になっているか確認し、店側の語りになっている箇所は自動的に修正する。修正した場合は「どの箇所をどう直したか」をユーザーに報告すること。

---

**最重要：読点（、）は最小限に抑える**
これがKikumiさんの文章の最大の特徴。長い文でも読点なしで書くことが多い。
生成した文章は必ず読点を見直し、なくせるものはすべて削除する。

**文体の特徴：**
- 短い文を重ねてリズムを作る
- 「です・ます」調を基本に、体言止めや断定文（「〜だ。」「〜がうれしかった。」）を自然に交える
- 一人称は「私」固定
- 女性的で温かみのある表現（「うれしいです」「感動しました」「〜が素敵でした」など）
- 読者への語りかけを自然に入れる（「ぜひ〜してみてください」「〜と思いませんか？」）
- 体験・感情を素直に言葉にする
- ！を適度に使う（多用しない）
- 他店・他社への批判・ネガティブ比較はしない

**構成のルール：**
- **冒頭に「答えリード」を1〜2行置く（AEO必須）**：記事の主旨・結論を太字で簡潔にまとめる。AIが150字以内に直接回答を抽出できるようにするための要件。その後に体験・情景の本文を続ける。
  ```markdown
  **〇〇は△△です。□□することで〜できます。**

  カウンターに腰を下ろして…（体験・情景の本文）
  ```
- **答えリードは厳格に150字以内**：AIが引用する際に150字を超えると切り捨てられる可能性がある。下書き生成後に必ず文字数を確認し超過していたら削る。GBCの店名・地名・URLは含めなくてよい（本文で補う）
  - **削り方の優先順位**：①GBCへの言及（「〜が私の場所です」等の締め文）→ ②副詞・接続詞・説明の重複 → ③2文目の後半。定義文（「〜とは〇〇のことです」）は削らない
  - **既存記事への適用**：新規作成時だけでなく、既存記事を編集するタイミングでも答えリードの字数を確認し150字超えなら修正する
- **FAQは検索口語体で書く（AEO・音声検索対応）**：「〜とは何ですか？」「〜はどこですか？」「〜を探しています」など実際に声で検索されるような自然な質問文にする。「〜について教えてください」より「〜はどうすればいいですか？」のほうが音声検索にヒットしやすい
- まとめセクションは任意（執筆開始時に確認）
- Q&Aは任意（執筆開始時に確認）
- 太字（`**強調**`）は適度に使う（多用しない）

- https://horikikumi.pages.dev/healthinfo/ の記事も文体の参考になる
